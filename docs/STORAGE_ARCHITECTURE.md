# 文件与存储架构

## 1. 设计目标

存储层需要同时满足：大文件、海量照片、断点续传、版本历史、用户内去重、配额统计、加密、校验、备份和未来应用授权。应用和客户端不能依赖服务器上的真实文件路径。

统一的是数据标识、权限和生命周期，而不是把所有内容塞入一个数据库。

## 2. 逻辑模型

```text
Vault（用户数据域）
├── Namespace（目录、相册或应用视图）
│   └── Entry（名称与层级关系）
├── File Object（文件逻辑对象）
│   └── Version（不可变版本）
│       └── Manifest（内容清单）
│           └── Chunk References
├── Blob Store（加密数据块）
├── Metadata Store（元数据与事务）
└── Derived Store（缩略图、索引等可重建数据）
```

### Vault

用户或未来家庭成员的隔离数据域，是去重、密钥、配额和访问控制的基本边界。不得跨用户使用可观察的全局去重。

### Entry

表示目录中的名称和层级。移动或重命名Entry不需要复制文件内容。同一File Object可以被不同相册或应用视图引用。

### File Object与Version

File Object代表一个长期文件身份；每次修改产生不可变Version。Version保存大小、媒体类型、时间、来源设备、完整内容摘要和Manifest引用。

### Manifest与Chunk

大文件按固定或内容定义策略切分为Chunk。Manifest按顺序记录Chunk标识、明文长度、密文长度和完整性信息。首版优先选择实现简单、边界清楚的固定分块，后续以真实数据测试是否需要内容定义分块。

## 3. 建议首版参数

- 默认Chunk大小：4 MiB；
- 小于4 MiB的文件使用单块；
- 上传会话具有明确有效期和最大总量；
- 客户端先计算文件摘要和分块摘要；
- 服务端在提交Version前验证所有Chunk存在且长度正确；
- 完整文件摘要用于恢复校验，不单独作为访问授权依据。

这些参数必须版本化，客户端不能假设永远不变。

## 4. 数据库存什么

关系数据库保存需要事务一致性的元数据：

- 用户、设备、应用和会话；
- Vault、Entry、File Object、Version和Manifest；
- Chunk引用、逻辑大小和实际占用；
- 上传会话、幂等键和同步游标；
- 分享、授权、撤销和审计事件；
- 快照、回收站和保留策略；
- 后台任务及其重试状态。

文件内容不作为大型字段直接塞入关系数据库。缩略图、全文索引和媒体分析结果必须可从原始数据重建。

## 5. 用户内去重

首版只在同一Vault内去重：

1. 客户端声明分块摘要；
2. 服务端返回缺失Chunk集合；
3. 客户端只上传缺失Chunk；
4. 服务端提交Manifest和Version时增加引用；
5. 删除Version只减少引用；
6. 引用归零且超过保留期后，后台垃圾回收才能删除Chunk。

去重查询本身可能泄露“某内容是否存在”。服务端必须先验证调用者属于该Vault，并对查询数量、速率和大小设置限制。

## 6. 加密模型

### 传输

所有设备到节点的连接使用现代TLS。设备绑定后还需要设备级密钥证明，不能仅依赖密码或家庭局域网位置。

### 静态数据

每个Vault拥有独立密钥层级；每个Chunk使用独立数据加密密钥和唯一随机数，通过经过认证的加密算法保护。数据密钥由Vault密钥封装，根恢复材料不长期明文保存在应用进程中。

### 现实边界

节点需要为已授权应用提供预览、索引或下载时，运行中的节点可能需要解锁相关数据。因此静态加密可以保护被盗硬盘和离线备份，但不能独自防御已经完全控制运行中节点的攻击者。应用隔离、最小权限、更新和审计仍然必需。

确定性加密有助于去重，却会增加内容确认风险。首版应把安全边界限制在单个Vault内，并在实现前完成专门的密码学设计评审。

## 7. 配额与空间账单

需要区分：

- 逻辑大小：用户目录中所有文件版本的总大小；
- 当前大小：当前可见版本的总大小；
- 历史大小：旧版本和回收站占用；
- 实际大小：去重和加密后实际占用；
- 派生大小：缩略图、索引和缓存；
- 备份大小：独立备份目标的占用。

上传前预留配额，提交成功后转为实际计量；失败或过期会话释放预留空间。不能依赖定期扫描文件系统作为唯一计费来源。

## 8. 一致性与崩溃恢复

Version提交采用两阶段语义：先上传不可见Chunk，再用单个元数据事务发布Manifest和Version。崩溃后可能留下孤立Chunk，但不能产生指向缺失Chunk的可见文件。

后台任务负责：

- 清理过期上传和孤立Chunk；
- 校验随机数据块；
- 重建派生索引；
- 执行保留与垃圾回收策略；
- 检查主存储与备份清单差异。

## 9. 备份不是同步

同步会传播删除和损坏，不能替代备份。节点至少支持：

- 本地快照或版本历史；
- 与主存储隔离的加密备份；
- 不可由普通节点会话立即删除的历史副本；
- 定期恢复抽检和完整性报告。

备份内容、元数据和密钥材料必须具有匹配的恢复版本；只有数据块而没有元数据或密钥不能构成可用备份。

## 10. 文件系统布局

物理布局是内部实现，不作为公共API。可采用类似布局：

```text
data/
├── blobs/<vault>/<prefix>/<chunk-id>
├── staging/<upload-session>/
├── derived/<vault>/
├── metadata/
└── quarantine/
```

客户端和应用只能通过对象ID和授权API访问数据，不能直接获知或操作上述路径。
