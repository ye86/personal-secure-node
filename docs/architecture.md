# PSN Node 公开架构说明

## 1. 定位

PSN Node 是一个以个人为信任中心的网络与数据运行环境。家庭服务器承担持续在线的身份、数据、授权、通信和恢复职责；电脑、手机、电视及物联网设备是可登记、可替换、可撤销的终端。

PSN 不计划从头实现通用操作系统内核。早期实现应复用成熟内核、虚拟化和容器技术，把主要工程投入放在个人身份、数据治理和应用权限边界上。

## 2. 信任模型

- 用户控制根身份和恢复方式。
- 家庭局域网不被默认信任。
- 域名是可读地址和发现入口，不是身份本身。
- 每台设备和每个应用具有独立身份。
- 应用、云端、中继和其他节点均不因其位置而自动可信。
- 根权限不提供给普通应用；高风险操作需要明确批准。

## 3. 主要组件

### 3.1 身份服务

负责根身份、设备登记、应用化名身份、会话凭证、撤销和恢复。主身份私钥不直接提供给应用；应用只获得限定对象、期限和用途的证明。

### 3.2 节点发现与连接

域名发布节点地址和身份声明。连接顺序原则上为直连、可信网络辅助穿透、端到端加密中继。中继只能获得完成转发所需的最少元数据。

### 3.3 个人数据层

统一管理数据目录、存储生命周期、来源关系、授权和审计，但不强制所有内容进入同一个物理数据库。

- 媒体和大文件使用对象存储；
- 联系人、消息和日历使用结构化存储；
- 设备遥测可使用时序存储；
- 搜索索引和缩略图属于可重建数据；
- 应用缓存和内部状态位于独立隔离空间。

同一用户范围内可以使用内容寻址减少重复数据，但必须保留独立权限、版本、引用计数和删除语义。

### 3.4 能力授权服务

授权至少应包含：主体、数据范围、操作、用途、目标接收者、有效期、次数、数据量和是否允许派生或转发。

读取权、修改权、网络访问权和数据发送权相互独立。应用不能因为能够读取数据且能够联网，就自动获得发送该数据的权利。

### 3.5 应用运行环境

应用分为三类：

1. 原生受控应用，通过 PSN 数据与身份接口运行；
2. Web/PWA 或跨平台应用，通过授权网关访问能力；
3. 旧有 Windows 应用和游戏，在隔离兼容环境中运行。

Windows 兼容环境是迁移工具，不是个人数据的信任根。它只接收用户明确挂载的数据视图，并受到存储、设备和网络策略限制。

### 3.6 审计与恢复

系统应让用户回答：哪个应用在何时读取、修改或发送了哪类数据；数据目前位于何处；哪些设备仍具有权限。审计记录本身也需要最小化、完整性保护和保留期限。

恢复设计必须覆盖设备丢失、服务器损坏、域名失效、密钥泄露和家庭断网，且不能依赖单个永久在线的第三方平台。

## 4. 数据外发边界

如果应用同时获得敏感明文和任意网络能力，系统无法可靠保证其不泄露数据。因此高敏感场景应优先采用以下方式：

- 应用默认无网络；
- 网络请求经过受控出口；
- 系统代替应用完成脱敏、加密和发送；
- 应用只接收完成任务所需的派生结果；
- 数据来源和派生关系随对象传播；
- 异常检测作为补充，而不是唯一安全边界。

## 5. 兼容与迁移

PSN 的推广顺序不是要求用户立即放弃 Windows：

1. 家庭节点配合 Windows 客户端和浏览器入口；
2. 管理个人文件、存档、身份批准和备份；
3. 将 Windows 应用放入分用途隔离环境；
4. 为新应用提供跨平台 SDK；
5. 在生态成熟后推出独立个人操作环境。

旧平台账号在过渡期作为外部身份和资产凭证存在。PSN 可以保存绑定关系并签发短期会话，但不能单方面替代游戏购买记录、在线封禁、排行榜或运营方规则。

## 6. 首个可验证闭环

项目的第一个重要里程碑是让两个节点完成：

1. 创建并安全备份个人身份；
2. 通过域名发现对方节点；
3. 验证对方长期身份并建立前向安全会话；
4. 发送消息和文件；
5. 将收到的数据登记到个人数据层；
6. 向演示应用授予有限读取能力；
7. 查看并撤销授权；
8. 从可信备份恢复节点。

在这个闭环通过威胁建模和测试之前，不扩大公网服务范围。

## 7. 明确不做的承诺

- 不宣称能够防御所有攻击者或所有零日漏洞；
- 不把 DNS、家庭路由器或单个云服务商视为绝对可信；
- 不让应用直接持有个人根身份；
- 不以自行发明密码算法作为创新点；
- 不把概念验证代码描述为生产级安全产品；
- 不为了统一而把全部数据塞入单一数据库。
