# 文件同步协议设计

本文定义首版需要满足的行为，不把当前v0.1密码学演示视为协议实现。

## 1. 同步对象

同步以不可变Version和有序变更日志为基础。每个设备维护自己的同步游标，不通过扫描服务器物理目录推断状态。

服务端变更事件至少包含：事件序号、对象ID、操作类型、新Version、父目录、发生时间和发起设备。事件序号只用于排序，不能代替安全校验。

## 2. 上传流程

```text
客户端申请上传会话
→ 声明目标目录、文件元数据、大小和幂等键
→ 服务端检查权限、配额和策略
→ 客户端提交Manifest草案
→ 服务端返回缺失Chunk
→ 客户端并发上传并可断点续传
→ 客户端请求提交Version
→ 服务端事务性发布并返回新游标
```

重复提交同一个幂等键必须返回原结果或明确冲突，不能创建多个不可区分的版本。

## 3. 下载流程

客户端先取得授权后的Version和Manifest，再并发下载Chunk，逐块验证并最终校验完整文件。临时文件校验成功后才原子替换本地目标。

## 4. 冲突处理

服务端不使用“最后写入者覆盖”静默丢弃数据。当两个设备基于同一旧Version修改时：

- 保留两个不可变Version；
- 创建可见冲突副本；
- 标注来源设备和时间；
- 允许用户选择、重命名或由支持的文件类型合并。

目录重命名和文件修改需要分别建模，避免把路径当作文件永久身份。

## 5. 删除语义

删除首先产生墓碑并进入回收站，不立即删除Chunk。墓碑随同步日志传播，包含删除发起者和保留截止时间。恢复创建新的可见Version或Entry事件，而不是抹掉审计历史。

永久删除需要更高权限，并受到快照、共享引用、合规保留和备份策略限制。

## 6. 离线与网络变化

- 上传会话可续期但不能无限存活；
- 每个Chunk请求可独立重试；
- 客户端使用指数退避和随机抖动；
- 从Wi-Fi切换到移动网络时遵守用户流量策略；
- 后台受限平台允许延迟，但必须显示最后成功同步时间；
- 客户端时钟只用于显示，冲突顺序以服务端事件和因果基线判断。

## 7. 安全要求

- 所有请求绑定设备身份、会话、Vault和具体能力；
- 上传声明的摘要不能代替服务端完整性验证；
- 下载URL短期有效并绑定对象、操作和接收者；
- 分享令牌以高熵随机值生成，服务端只保留必要的验证材料；
- 防止路径穿越、压缩炸弹、超大Manifest和Chunk数量滥用；
- 元数据、文件名和缩略图也属于敏感数据；
- 外部上传只进入隔离区，不能触发任意服务端解析器。

## 8. 协议演进

API和对象格式需要显式版本。服务端公布支持范围，客户端协商共同版本；安全相关能力不能在无提示情况下向更弱方案降级。未知字段应按规范保留或拒绝，不能由各客户端随意解释。
